The Ground Truth: UNSW-NB15 Dataset

To train our Machine Learning gatekeepers, we needed a massive, realistic dataset. We utilized the prestigious UNSW-NB15 cyber security dataset, which contains pure, organic network traffic captures.

Vast Scale

The dataset contains over 175,000 distinct data points (network flows) in its core training set, representing a massive array of both benign enterprise traffic and modern cyber attacks.

9 Attack Categories

The data is flawlessly categorized into distinct vectors, including: DoS, Exploits, Reconnaissance, Backdoors, Fuzzers, Analysis, Shellcode, and Worms.

No Synthetic Fakes

We strictly enforced a "Purity Rule". We did not forge fake data points using tools like SMOTE. We simply amplified the real, organic signatures so the AI learns what an actual attack looks like, not a hallucination.

Data Engineering: The Math Behind the Packets

A network flow generates dozens of chaotic metrics. To make this digestable for our ML models in real-time, we compress them into exactly 9 core mathematical features.

The Raw Features

We capture raw statistics like:

  • spkts & dpkts (Packets sent/received)
  • sbytes & dbytes (Bytes sent/received)
  • dur (Flow duration)

Logarithmic Compression

Why we do it: A sudden DoS attack could send 10,000x more bytes than normal, breaking the ML math.

By applying a "Logarithmic Transformation", we squeeze these massive numbers down into a predictable, smooth scale that the AI can instantly understand without crashing.

Behavioral Ratios

Why we do it: Raw packet counts fluctuate. Ratios don't.

We divide sent bytes by received bytes. This exposes the behavior of the attack. (e.g., A Port Sweep sends lots of data but receives almost nothing back!).

The Machine Learning Gatekeeper

Before any Generative AI gets invoked, every single network flow must pass through a two-stage Random Forest pipeline. Why Random Forest? Because it is incredibly fast at real-time packet inspection and highly resilient to noisy data.

Stage 1: The Bouncer (Binary)

A simple Yes/No model. Determines if the traffic is Normal or an Anomaly.

  • Accuracy: 94.6%
  • Precision (Attack): 96.8%

This filters out 99% of normal background traffic so we don't flood our expensive LLM APIs!

Stage 2: The Detective (Multi-Class)

If an anomaly is detected, this model categorizes the specific attack vector (DoS, Exploit, Reconnaissance).

We trained this with 150 parallel "decision trees" capping at a depth of 15 to ensure it doesn't overthink simple patterns.

Overcoming ML Roadblocks

Training an AI in a vacuum is easy. Deploying it in a live Cyber Range is hard. Here are two massive engineering challenges we solved:

Header Offsets

Problem: Our synthetic Scapy attacks were missing 34 bytes of raw Ethernet headers compared to the original dataset. The AI was rejecting them.

Fix: We mathematically offset our live payload generation to perfectly mirror the Layer-2 physical captures of the UNSW-NB15 dataset.

Dense Overlap

Problem: In the raw data, "Backdoors" and "DoS" attacks look mathematically identical, causing the AI to confuse them.

Fix: We abandoned the "median" averages and targeted the specific, distinct mathematical "mean" of the Backdoor attacks, carving out a clean boundary the AI could easily spot.

Agentic AI Swarm (LangGraph)

When the ML model triggers, the flow metadata is handed off to a multi-agent AI pipeline powered by Gemini 2.5 Flash. The agents reason sequentially about the threat.

1. Triage Agent

Evaluates the raw network statistics and the ML prediction. Determines the immediate severity of the flow.

2. Context Agent

Cross-references the Destination IP against our mock Asset Registry to determine the business impact (e.g., hitting an HR database vs a DMZ honeypot).

3. Response Agent

Weighs the severity against the context and makes a definitive Go/No-Go decision on mitigation.

Autonomous Mitigation

The AI isn't just an alert generator—it has teeth. The Response Agent is equipped with a Python @tool that allows it to physically alter the network state.

The Block Execution

If the AI decides the threat is critical, it dynamically invokes the mitigation tool, dropping the attacker's connection at the firewall level:

os.system(f"iptables -A FORWARD -s {attacker_ip} -j DROP")

We also implemented a global cooldown debounce and an in-memory IP blacklist to prevent volumetric DDoS attacks from repeatedly triggering expensive LLM token usage.
