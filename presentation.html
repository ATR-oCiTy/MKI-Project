Integration Goal
StayWell is deploying Chatwoot — an open-source omnichannel customer engagement platform — to consolidate guest communications from public website chat, email, and WhatsApp Business API into a single dashboard used by Customer Care Agents.
Technology Stack
APP Chatwoot Ruby on Rails · Open Source
DB PostgreSQL Guest PII · Chat Logs · Credentials
Q Redis Background job queue
PRX Nginx Reverse proxy · TLS termination
CTR Docker Self-hosted containerised deployment
API WhatsApp Business API External · Meta · HMAC-signed webhooks
Why Self-Hosted?
Total control over personal data · GDPR compliance · No third-party data processor agreements required

System Architecture — Trust Boundaries
13 DFD elements across 3 trust zones · Threat modelling applied at every trust boundary crossing
Untrusted External Zone
EL01 Guest Customer EL02 Customer Care Agent EL03 WhatsApp Business API
All external entities are untrusted sources. Webhook signatures must be validated.
↓ EL08 Untrusted HTTPS  ·  EL09 Authenticated HTTPS  ·  EL10 API Webhook Traffic
External Edge Boundary (DMZ)
EL04 Nginx Reverse Proxy EL05 Chatwoot Core Application
TLS termination at Nginx. Application not directly reachable from outside. EL11 internal proxy flow.
↓ EL12 TCP/IP (SQL Data)  ·  EL13 TCP/IP (Cache/Jobs)
Internal Restricted Boundary
EL06 PostgreSQL Database EL07 Redis Queue
Data stores must never be publicly reachable. Redis restricted to localhost only.
Key Trust Boundary Crossings
EL08 Public chat → Nginx: untrusted, unauthenticated, input must be sanitised
EL10 WhatsApp webhook → Nginx: external, must be HMAC-verified before processing
EL11 Nginx → Chatwoot: internal proxy; application must not be directly reachable
EL12 Chatwoot → PostgreSQL: encrypted; parameterised queries mandatory to prevent SQL injection
EL13 Chatwoot → Redis: restricted to localhost; no external exposure permitted

Use Case Analysis
What the System Does — and What Security Each Interaction Requires
Three use cases defined · Each scoped to specific actors and security objectives · Directly drives Security Requirements
UC01
C I A
Omnichannel Guest Intake
A hotel guest submits a support request via the public website chat widget. Chatwoot routes it as a new ticket in the agent dashboard. The guest is unauthenticated — no login, no verification required.
SH06 Guest SH03 Agent EL01 → EL08
Security significance
Fully unauthenticated public entry point — the highest-exposure surface in the system. Three High-rated threats originate here: TR02 (XSS), TR03 (SQLi), TR05 (DoS). All input must be sanitised before reaching the database. Drives SR05, SR07.
UC02
C I
Agent Ticket Management
An authenticated Customer Care Agent reads, replies to, and closes guest support tickets through the Chatwoot dashboard via an authenticated HTTPS session (EL09). Agents handle guest PII directly.
SH03 Agent EL02 → EL09
Security significance
Agent sessions must be protected against hijacking (TR02, TR09). All agent actions must be logged for GDPR accountability (TR08, ASVS 8.3.7). Horizontal access between agent accounts must be prevented (SUS02 — IDOR). Drives SR05.
UC03
C I
User Administration
The IT Ops Manager creates and manages agent accounts, configures permissions, and updates webhook integrations. Requires administrator role — the highest privilege level in the system.
SH04 IT Ops Admin EL02 → EL09
Security significance
Admin actions define the entire access control model. Only the admin role may modify webhook configurations (SR06). Privilege escalation (TR06) here grants god-mode access to all assets. Mass assignment (EVS01) is the specific attack vector. Drives SR05, SR06.
UC01 → SR05, SR07 UC02 → SR05 UC03 → SR05, SR06 Security requirements derived next — from these use cases + stakeholders + interfaces

Conclusion
Security Posture Summary
Full traceability from stakeholders through to threat decisions · GDPR compliance built into the model
GDPR Article 33 — Primary Compliance Driver
72-Hour Breach Notification Obligation
StayWell, as a GmbH processing guest personal data under German jurisdiction, is supervised by the LfDI / BfDI. Five of the eight damage scenarios (DS02, DS03, DS04, DS06, DS08) independently trigger a mandatory breach notification to the supervisory authority within 72 hours. This drove the High protection classification for Customer PII (AS01), and the Very High classification for agent credentials (AS03), integration data (AS04), and container config (AS06), shaping the severity of six of the ten identified threats.

Key Findings
01
The public chat widget is the highest-risk entry point. It enables three of the six High-rated threats (TR02 XSS, TR03 SQL Injection, TR05 DoS) from unauthenticated external access with no authentication barrier.
02
Insider threat and repudiation are underestimated in typical chat platform deployments. TR06 (session escalation) and TR08 (repudiation) together cover DS04 — the most damaging scenario — because there is no audit trail and no post-incident forensic capability without SR10 in place.
03
Container misconfiguration is a single point of total failure. TR10 (Docker Compose credential exposure) leads to DS08 — full compromise of all six assets in a single read operation. SR16 (runtime secrets injection) is the critical control.
04
TR07 is correctly Accepted — not all risks require mitigation. The WhatsApp regional outage is outside StayWell's control boundary. The Accept decision is justified because no internal data is compromised and StayWell has no contractual leverage over Meta infrastructure uptime.
